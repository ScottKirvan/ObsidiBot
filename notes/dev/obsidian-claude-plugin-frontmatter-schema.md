# Obsidian Claude Plugin — Frontmatter Schema Specification

**Version:** 0.1  
**Status:** Draft — finalize before implementing `FrontmatterGuard.ts` and `ContextManager.ts`

---

## Overview

The plugin reads a `claude` key in YAML frontmatter to determine how Claude should interact with individual notes. This schema is the interface between the user's vault and the plugin's behavior — it should be stable, intuitive, and additive (new fields can be added without breaking existing ones).

All fields are optional. A note with no `claude` frontmatter key behaves normally with no restrictions or special treatment.

---

## Full Schema

```yaml
---
claude:
  context: always | never | session    # Context inclusion behavior
  readonly: true | false               # Prevent Claude from writing to this file
  protect: true | false                # Alias for readonly (pick one convention)
  instructions: "string"              # Per-note instructions injected when file is accessed
  priority: high | normal | low        # Hint for context ordering when multiple notes are pinned
---
```

---

## Field Reference

---

### `context`

**Type:** string enum  
**Values:** `always` | `never` | `session`  
**Default:** (absent — standard behavior)

Controls whether this note is automatically included in Claude's context.

#### `context: always`

The note is injected into every session automatically, without the user having to pin it manually. Treated as a permanent pin.

**Behavior:**
- Included in the initial context payload on every session start
- Appears in the plugin's "Active Context" display as a vault-managed pin (distinct from user-managed pins)
- Survives session changes — present in new sessions started with `+`
- The plugin scans for `context: always` notes at session initialization and includes their content

**Use cases:**
- Master index or table of contents
- Style guide or writing conventions
- Cast of characters in a fiction project
- Canonical taxonomy or ontology
- Project glossary

**Example:**
```yaml
---
claude:
  context: always
  instructions: "This is the master project index. Reference it when navigating the vault structure."
---
```

---

#### `context: never`

The note is excluded from Claude's access entirely. The plugin blocks any tool call that would read this file.

**Behavior:**
- If Claude attempts a read tool call on this file, the plugin intercepts and blocks it
- Plugin returns a message to Claude: `"[filename] is marked as restricted. Do not attempt to access it."`
- Claude relays the restriction to the user
- The file still appears in the folder tree injection (Claude knows it exists but cannot read it)
- If user explicitly asks Claude to read the file, Claude will attempt a read → plugin blocks → Claude explains it cannot access the file

**Use cases:**
- Sensitive personal notes (passwords, private journals, financial records)
- Notes not relevant to any Claude workflow
- Files being actively edited that shouldn't be touched mid-session

**Example:**
```yaml
---
claude:
  context: never
---
```

**Edge cases:**
- A note marked `context: never` AND `readonly: true` — `never` takes precedence (Claude can't read it, so write protection is moot but harmless to include)
- If the user configures a note as `context: never` but also pins it manually via the UI — `never` wins, UI pin is ignored, and the user is shown a warning

---

#### `context: session`

*(Reserved for future use — not implemented in v1)*

Intended to mark notes that should be available to Claude within a session once referenced, but not auto-injected. Effectively the default behavior. Reserved to allow explicit opt-in to "normal" behavior for notes that might otherwise match an auto-include rule in a future version.

---

### `readonly`

**Type:** boolean  
**Default:** `false`

Prevents Claude from writing to or modifying this file. Claude can still read it.

**Behavior:**
- Plugin intercepts any write, edit, or delete tool call targeting this file
- Write is blocked before execution
- Plugin sends Claude a message: `"[filename] is marked as read-only. You may read this file but cannot modify it. Suggest changes in the conversation instead."`
- Claude's suggested edits are shown in the chat but not applied
- The note remains readable and referenceable

**Use cases:**
- Canonical reference documents (taxonomies, style guides, templates)
- Completed and archived notes that should not be reopened
- Notes maintained by external tools or processes
- Any file where accidental modification would be hard to recover

**Example:**
```yaml
---
claude:
  readonly: true
  instructions: "This is the canonical tag taxonomy. If you think a tag should be added or changed, say so in the conversation — do not edit this file."
---
```

---

### `protect`

**Type:** boolean  
**Default:** `false`  
**Alias for:** `readonly`

Identical behavior to `readonly`. Provided as an alternative field name because "protect" is more intuitive for some users than "readonly."

**Implementation note:** If both `readonly` and `protect` are present in the same frontmatter, either `true` value triggers the restriction. They are logically OR'd.

```yaml
# These are equivalent:
claude:
  readonly: true

claude:
  protect: true
```

---

### `instructions`

**Type:** string  
**Default:** (absent — no per-note instructions)

A string of instructions that is injected into Claude's context whenever this file is read or referenced during a session.

**Behavior:**
- When Claude reads this file via a tool call, the plugin appends the instructions to the tool result before returning it to Claude
- Format: `"[File content]\n\n---\nNote instructions: [instructions]"`
- Instructions are also included in the initial context if the note is pinned (`context: always`) or manually added to session context
- Instructions survive across the session once the file has been read

**Use cases:**
- "This is a raw meeting transcript. Do not reformat it."
- "This is the canonical source of truth. Treat it as authoritative."
- "When editing this note, preserve the existing structure exactly."
- "This document is a template. When referencing it, use it as a model but create new files — don't edit this one."
- "Always check for consistency with [other-note.md] before making changes here."

**Length guidance:** Keep instructions concise — under 150 words. Long instructions bloat context on every read. If you need more than a paragraph, consider whether it belongs in the vault context file instead.

**Example:**
```yaml
---
claude:
  readonly: true
  instructions: "Canonical project taxonomy. Reference when organizing notes or suggesting tags. Propose additions in conversation but never edit directly."
---
```

---

### `priority`

**Type:** string enum  
**Values:** `high` | `normal` | `low`  
**Default:** `normal`  
*(Reserved for future use — not implemented in v1)*

Provides a hint for ordering when multiple notes are injected into context simultaneously and a prioritization decision is needed. Higher priority notes are injected first (closer to the beginning of the context window, where attention weight is higher in some model behaviors).

## Compound Examples

**Protected reference document, always present:**
```yaml
---
claude:
  context: always
  readonly: true
  instructions: "This is the master project taxonomy. Always consult it when categorizing notes or suggesting structure changes. Never modify it — propose changes in conversation."
---
```

**Meeting transcript (session use only, with handling instructions):**
```yaml
---
claude:
  instructions: "Raw meeting transcript. Extract action items and key decisions when asked, but do not reformat the note itself."
---
```

**Sensitive note, fully excluded:**
```yaml
---
claude:
  context: never
---
```

**Style guide, always available, protected:**
```yaml
---
claude:
  context: always
  protect: true
  instructions: "House style guide. Apply these conventions to all writing you produce or edit in this vault."
---
```

**Template note:**
```yaml
---
claude:
  readonly: true
  instructions: "This is a template. When the user asks you to create something based on this, create a new file using this as a model. Do not edit this file."
---
```

## Plugin Behavior Summary Table

| Field combination | Claude can read? | Claude can write? | Auto-injected? |
|---|---|---|---|
| (no claude key) | Yes | Yes | No |
| `context: always` | Yes | Yes | Yes, every session |
| `context: never` | No | No | No |
| `readonly: true` | Yes | No | No |
| `protect: true` | Yes | No | No |
| `context: always` + `readonly: true` | Yes | No | Yes, every session |
| `context: never` + `readonly: true` | No | No | No (`never` dominates) |

## Implementation Notes

### Reading Frontmatter

Use Obsidian's `metadataCache` API — do not parse YAML directly:

```typescript
const cache = this.app.metadataCache.getFileCache(file);
const claudeMeta = cache?.frontmatter?.claude as ClaudeFrontmatter | undefined;
```

The cache is always up to date and doesn't require reading the file.

### TypeScript Interface

```typescript
interface ClaudeFrontmatter {
  context?: 'always' | 'never' | 'session';
  readonly?: boolean;
  protect?: boolean;
  instructions?: string;
  priority?: 'high' | 'normal' | 'low';
}

function isReadonly(meta: ClaudeFrontmatter): boolean {
  return meta.readonly === true || meta.protect === true;
}

function isExcluded(meta: ClaudeFrontmatter): boolean {
  return meta.context === 'never';
}

function isAlwaysPinned(meta: ClaudeFrontmatter): boolean {
  return meta.context === 'always';
}
```

### Vault Scan at Session Start

On session initialization, scan all files for `context: always` and `context: never`:

```typescript
async function scanVaultFrontmatter(app: App): Promise<{
  alwaysPinned: TFile[];
  excluded: TFile[];
}> {
  const alwaysPinned: TFile[] = [];
  const excluded: TFile[] = [];

  for (const file of app.vault.getMarkdownFiles()) {
    const cache = app.metadataCache.getFileCache(file);
    const meta = cache?.frontmatter?.claude as ClaudeFrontmatter | undefined;
    if (!meta) continue;

    if (isAlwaysPinned(meta)) alwaysPinned.push(file);
    if (isExcluded(meta)) excluded.push(file);
  }

  return { alwaysPinned, excluded };
}
```

The excluded list is passed to `FrontmatterGuard` so it can block tool calls efficiently without re-reading frontmatter on every call.

### Intercepting Tool Calls

Before passing any tool result back to Claude, check the guard:

```typescript
function guardToolCall(
  toolName: string,
  filePath: string,
  excluded: Set<string>,
  readonly: Set<string>
): { allowed: boolean; reason?: string } {
  if (excluded.has(filePath)) {
    return { allowed: false, reason: `${filePath} is marked as restricted and cannot be accessed.` };
  }

  const isWrite = ['write', 'edit', 'delete', 'move', 'rename'].includes(toolName.toLowerCase());
  if (isWrite && readonly.has(filePath)) {
    return { allowed: false, reason: `${filePath} is marked as read-only. You may suggest changes in the conversation but cannot modify this file directly.` };
  }

  return { allowed: true };
}
```

## Schema Stability Commitment

Once v1 ships, treat this schema as a public API. Users will add this frontmatter to their notes and expect it to keep working across plugin updates. Breaking changes (renaming fields, changing behavior of existing values) require a migration path and clear changelog entry.

Additive changes (new fields, new enum values) are non-breaking and can be introduced in any version.
